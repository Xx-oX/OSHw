程式執行時間：

造成實際比參數大的原由是因為input.txt為文字檔，不能夠很好的控制大小（string 的大小隨數字量而定）

  • 1 Gb(實際 1.2Gb)：

    • 執行./sort 98.3676
    • 執行./sort 1 1 (軟體優化) 89.1583

  • 8 Gb(實際 9.4Gb)： 780.668秒

  • 16 Gb(實際 19Gb)： 2140.04秒

程式開發與使用說明：

使用說明：

  • rng
    • 功能：產生亂數檔
    • 指令：./rng arg1 arg2
      • arg1 : 1 -> Kb
        2 -> Mb
        3 -> Gb
      • arg2 : 倍數
      • 大約產生出大小為 arg2*1024^arg1 bytes 的檔案
  • sort
    • 功能：執行外部排序
    • 指令：
    • ./sort
      • 預設的執行方式（Gb）
    • ./sort arg1
      • 指定input檔名的執行方式
    • ./sort arg1 arg2
      • arg1 : 1 -> Kb
        2 -> Mb
        3 -> Gb
      • arg2 : 倍數
      • input.txt的大約大小（arg2*1024^arg1 bytes）
      • 自動算出比較適合的參數
    • ./sort arg1 arg2 arg3
      • arg1 : 將input.txt分成數個大小為arg1的小檔案
      • arg2 : 在進行merge時，一個run的大小
      • arg3 : 輸出緩衝大小
      • 大約佔用RAM大小為arg2 * (sizeof(innput.txt)/arg1) + arg3 bytes
      • 手動給予參數

原理：

  • 分割方式：將input.txt分成數份，每份皆進行內排序，再存在不同的二進位檔（減少使用空間）裡。
  • 排序方式：
    • 內排序：使用c++函式庫<algorithm>中的sort函數（依照資料可能為quick sort 或 insertion sort）
    • 外排序：利用單層的K-way merge sort，一次讀取K個runs，每次將最小的加到output buffer，當buffer滿時再寫入output.txt。當run裡面所有的資料都被加到output buffer時，再從外部檔案讀入剩餘的（來確保不會使用超過RAM大小的記憶體空間）。

效能分析報告：

對硬體效能的優化：

  • 因為8Gb在跑得時候會吃滿記憶體，此時再做紀錄必定當機（已經經歷無數回）所以部份截圖只用小一點的來示範。
  • 程式主要分成divide()跟merge()兩個部份，divide的部份很吃cpu的資源，而merge的部份很吃RAM。
  • 利用top, perf等工具來實時監控電腦的狀況（效能）
  • 因為divide()能在優化的部份不是很多（只能盡量減少I/O的次數來增加效能），所以較多的觀察放在merge()上面。
  • merge()的時間複雜度為 O(kn) 其中k為run的數量，所以降低k的值可以有效提昇效能。
  • 藉由減少要merge的page數，來降低在merge()中cpu所要執行的instruction數，還有branch數，藉此來減少branch-misses，達成更好的效能。（見前述執行時間1Gb測資的圖）
  • 藉由perf可以發現，不管是branch instruction的數量，還是cpu-cycle，亦或是instructions的數量，前幾名都是有關於std::vector的相關操作。這點超乎我的預期，但有限於時間的關係，這方面沒有來得及做修改。但如果將一些c++ stl的容器改用c的方式實做，對效能的提升可想而知。
    • perf報告的截圖，依序為branch-instructions, branch-misses, cpu-cycles, instructions，可以發現前幾名都是跟vector有關的操作。



  • 結論：
    • divide()：增加一個分檔的大小來減少分檔的數量，藉此降低I/O瓶頸。
    • merge()：增加一個run的大小來減少run的數量，藉此來減少每一次要比較的數字量（還有降低cache miss的發生）。
    • 可以試著將高階的資料結構利用更有效率的方式來實作：比如c++的vector, string或許可以用int*,char*來達到更高的效率。
    • 低階語言在硬體的效能優化上面有很大的優勢。

同時運行時的觀察：

  • 因為同時跑太大的檔案很容易死當，所以我用4個1Gb的來做測試。
  • 主要的時間都是花在divide，畢竟2way小檔案merge很快。以下做個統計及比對
  • 同時跑4個的秒數：
func test1 test2 test3 test4
divide() 144.579 157.221 154.568 131.817
merge() 41.5094 45.5824 45.0101 41.8427
total 186.089 202.804 199.578 173.66
  • 只跑一個的秒數
func test1
divide() 76.8127
merge() 25.3295
total 102.214
  • 很明顯當4個程式都在搶cpu資源時，效能降低很多（時間幾乎2倍），這邊雖然不是很懂為何4支程式可以同時用100%的cpu(甚至超過)，或許是利用了不同的cpu。
  • 另一個可以預見的問題是RAM的分配，想像一下如果是跑4個8Gb的，估計會直接把RAM的空間吃光。
  • 不過如果運用multi-thread的方法的話，應該可以提升不少效能。

結論

  • 所以綜合以上的觀察，我認為一個作業系統的設計應該要提供：

    • 記憶體保護機制：不讓RAM被吃光而導致不可預期的錯誤
    • 妥善分配執行緒：達到高效率的多工
    • 適當的壓榨資源：在合理的範圍內，盡量讓每個程式可以用到100%的資源
    • 有效率的排程：如果資源實在不夠，利用排程讓每個程式的效能最大化
