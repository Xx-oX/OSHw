程式開發與使用說明：

使用說明：

// Compile
$ g++ -O0 -o transformer transformer.cpp -lpthread
// Run
$ ./transformer [threads]

編譯
  • rng
    • g++ -o rng rng.cpp
  • transformer (主要的程式)
    • g++ -O0 -o transformer transformer.cpp -lpthread
執行
  • rng
    • 功能：產生亂數檔(.csv)
    • 指令：./rng
      • 產生出大小約為1Gb的檔案
    • 指令：./rng arg1
      • 產生總共arg1行（每行20個數字）的檔案
  • transformer
    • 功能：將input.csv轉換成output.json
    • 指令：./transformer
      • 利用4個thread來執行轉換
    • 指令：./transformer arg1
      • 利用arg1個thread來執行轉換

流程

  • 讀檔
    • 先遍歷一次input.csv來獲得總行數，並初始化arr_input
    • 再重新讀一遍檔案，並存入arr_input裡面
  • 轉換
    • 利用多個thread來平行處理
    • 將arr_input[i]的內容轉換成json的格式，並存在json_strings的vector裡面
  • 寫檔
    • 將json_strings裡的string照順序寫入output.json中

效能分析報告：

讀檔

  • 本來讀檔案也是採multi-thread的方式來實作，但是因為檔案才1Gb，用multi-thread花的時間跟空間的資源都比直接一次讀完還多，時間大概與thread數成正比，故把讀檔改成一個thread直接完成。
  • 處理時間約在7～8秒

轉換

  • 程式的主要部分，也是唯一採用multi-thread的階段。起初裡面對arr_input的讀取是不連續的，導致cache-misses會隨著thread數大幅增加。於是我改成先把這個thread會使用到arr_input的部分先用pointer存起來，來嘗試降低cache-misses。

    • 本來的code
    











        for(int i=offset; i<offset+size; ++i, ++c){
        string str = "    {\n";
        for(int j=0; j<19; ++j){
            str += ("        \"col_" + to_string(j+1) + "\":" +
                to_string(arr_input[i][j]) + ",\n");
        }
        str += ("        \"col_" + to_string(20) + "\":" + 
            to_string(arr_input[i][19]) + "\n");
        if(i != lines_input-1) str += "    },\n";
        else str += "    }\n";
        json_strings[i] = str;
    }

    • 改進的code（利用perf可以發現cache-misses有明顯的降低）
    












        int **arr_p = &arr_input[offset];
    for(int i=0; i<size; ++i, ++c){
        string str = "    {\n";
        for(int j=0; j<19; ++j){
            str += ("        \"col_" + to_string(j+1) + "\":" +  
                to_string(arr_p[i][j]) + ",\n");
        }
        str += ("        \"col_" + to_string(20) + "\":" + 
            to_string(arr_p[i][19]) + "\n");
        if(i != lines_input-1) str += "    },\n";
        else str += "    }\n";
        json_strings[offset+i] = str;
    }

  • 處理時間：

    • 1個run表示處理一行（20個數字）

    • 總時間 ＝ max{一個run的時間 * run數}

      thread數 1 2 4 8
      1run的時間(μs) 11 24 50 150
      run數 4888888 2444444 1222222 611111
      總時間（s） 55 62 64 96

  • 發現thread越多，效率反而越差。有可能是因為處理的資料量不夠大，導致overhead的發生。

    • 但是藉由top的觀察，可以發現cpu使用率跟memory是跟thread數成正比的。
      cpu使用率 ＝ thread數 * 100% （表示multi-thread確實有成功）

      4 threads:

    • 藉由perf的觀察，發現cache-misses以及branch-misses都會隨著thread數增加而增加，或許是導致時間變長的原因。

  • 但是在windows作業系統中用mingw的lib編譯並執行，發現在thread數小於8時，效能的確是有跟著thread數增加的。
    或許這也跟不同作業系統處理thread的方法有關。

    • 可以發現轉換的部分所花的時間是隨著thread數增加而減少的。

寫檔

  • 如果用multi-thread的方式，因為順序的問題，勢必要使用critical section來保證各thread的順序。但是這樣其實就跟直接一次寫完一樣意思，甚至花更多時間在等待（thread被lock的時間）。
  • 處理時間約在5～6秒

結論

  • 因為讀寫檔案都是一次跑完，沒有開多執行緒，所以對效能影響最大的就是轉換的部分。
  • 本來以為thread越多，效能就越好。但事實證明還是要根據任務的性質來判斷，並不是越多越好。
